## Topics discussed
### Staff permission / access management (granular, module-wise)
- Senior discussed the expected **staff access** behavior:
  - Staff-level access should reflect exactly which roles/permissions are assigned to that staff.
  - There is an existing ‚Äústaff wise access‚Äù concept in the system, and permission assignment must map correctly into that.

### Group vs Admin vs Team/Department (correct access scoping)
- Senior explained the selection model expected for permission assignment:
  - If access is assigned at **group level**, then the staff should receive broader admin/administration access (as per system design).
  - If access is assigned at **admin level**, then the staff should see modules accordingly (admin expected to see all modules).
  - If access is assigned to a **specific team / specific department**, then the system should grant only that scoped access (granular).
- Current issue discussed:
  - The logic for ‚Äúspecial department/team access‚Äù appears to be **inverted** (group/admin assignment affects more than intended).
  - Senior asked for correction so that:
    - Admin gets administration scope correctly
    - Specific team/department constraints are respected.
- Senior also questioned UI/UX necessity of a ‚Äúradio button‚Äù approach:
  - After selecting `team`, the UI should not unintentionally force access to all related scopes; it should depend only on the selected scope.
