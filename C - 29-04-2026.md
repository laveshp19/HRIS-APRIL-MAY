## Topics discussed
### Leave summary output contract (what UI should receive)
- Senior clarified that **leave summary should show only the leave-balance output**, not the full/expanded leave application grid.
- Required behavior:
  - Leave summary output should include `StaffID` and staff display (name under the staff).
  - **Approval-related fields (approved by / approver name)** should be omitted from the “leave summary” view (even if they are available from the backend).
  - The “leave application” section below (if shown in the UI) should use the **actual period** (From/To dates), but the leave-application dataset itself should not be embedded inside the leave-summary response (as per discussion).

### Fix: doubled leaves for a specific StaffID (example: `108`)
- Senior raised a concern that for a specific staff (mentioned as **ID 108**):
  - leave counts are appearing **double**.
- Suspected cause: when staff is added/removed via **policy bundle** workflows, previous mappings/records might not be fully cleared, so leave balance/leave assignment is being applied more than once.
- Suggested check/verification:
  - Delete existing mapping for `StaffID = 108`
  - Then re-add it
  - Re-check leave summary results to confirm duplication is resolved.

### Where “Get applicable bundle” logic should live
- Senior asked to ensure **Get applicable bundle by staff ID** logic is handled from the correct controller (HRIS policy bundle controller) so the response is consistent with what UI expects.

### Leave application output contract (including raised/approved data)
- Senior explained that the “leave application” output should include:
  - `request raised by` (employee who raised)
  - `approved by` (reporting manager)
- This corresponds to the project stored procedure returning the raised/approved-by fields for leave applications.

### “Enabled by HR” leave types (example: maternity leave)
- Senior discussed a business rule:
  - HR may define leave types as **enabled by HR** (example given: **maternity leave**).
  - When adding staff into a policy bundle, the **enabled-by-HR items should NOT automatically appear** in the staff’s applicable leave list.
  - They should appear only after HR process activation (then it becomes visible on staff profile / applicable bundle view).
- Senior asked to implement this using the correct parameterization approach (enabled-by-HR should be treated as a parameter/condition for display).

### Year-wise nature
- Senior emphasized the feature is **year-wise** (leave summary and leave application data must depend on `AcademicYearID`).
