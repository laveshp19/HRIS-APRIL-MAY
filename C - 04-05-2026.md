## Topics discussed
### Attendance punch ingestion + triggers
- Senior demonstrated the end-to-end logic for handling staff biometric punch data:
  - Punch records come from HR attendance/device punches (Senior referred to the device/data source and device log mapping).
  - A trigger/automation is applied on the source punch table (Senior mentioned building a similar “detail table” in the local DB, then inserting into it).
  - Another trigger updates the main attendance structure in the system.

### IN/OUT mapping for first & subsequent punches
- The attendance logic described:
  - First punch is mapped into the main “IN” slot (transcript used “IN0”).
  - Second punch is mapped into the main “OUT” slot (transcript used “OUT0”).
  - The system supports multiple in/out cycles (Senior described provisioning for up to ~10 cycles), where additional punches update subsequent IN/OUT slots.
  - Multiple punches change counts and credit/deduction outcomes.

### Credit / deduction computation and where it flows
- Senior explained that:
  - Credit/deduction depends on the in/out differences and scheduler/day aggregation.
  - Deductions ultimately impact the leave deduction dataset (transcript referred to “leave deduction record”).

### Regularization flow (final approved times)
- Senior explained regularization handling:
  - Regularization is applied (by professor/school workflow) and then approved.
  - When approved, **final in/out values** are stored and used for credit decisions.
  - Machine punch vs regularization times are kept as separate concepts so the system can identify mismatches/identify the scenario when deductions/credits are off.
  - Senior also discussed the admin-side process of different slips (official vs personal work) and that TL approvals drive subsequent regularization updates.

### Temporary shift logic (when it is applied)
- Senior described how temporary shift affects calculations:
  - From policy bundle, start/end shift and relaxation windows are used by default.
  - If a **temporary shift** exists for that employee/day, it should **overwrite** the shift and relaxation values.
  - Important behavior: temporary shift checks happen only on **the first punch** (so second punch doesn’t re-check temporary shift because state is already captured).
  - If temporary shift logic is not ready yet, keep trigger behavior limited to regular policy bundle values.

### Flags: late/early and route-specific exceptions
- Senior covered how flags are computed and how exceptions apply:
  - “Is late in” / “Is early out” flags are determined by comparing punch times with relaxation/cutoff times.
  - Senior discussed route-based / transport-based logic:
    - “Own vehicle list” presence influences deduction behavior.
    - “Gatekeeper entry” presence influences deduction behavior.
    - Example described: late + own vehicle may avoid deduction depending on gatekeeper entry state (and vice versa, if gatekeeper entry is missing).

### Leave application integration: block if punch exists
- Senior highlighted a critical business rule for leave application:
  - If the employee has **any punch on the day they are applying leave**, then leave application should be blocked.
  - Backend behavior expected:
    - Detect punch exists
    - Remove/adjust punch data via backend (as discussed)
    - Then apply **full day leave**.
  - Edge cases mentioned:
    - Employee may punch in and then leave early (~15–20 minutes scenario).
    - Gate arrival quick scenario (punch exists but employee leaves immediately).

### Policy bundle prerequisites + UI verification
- Senior emphasized that triggers depend on policy bundle correctness:
  - Each employee must have correct policy bundle shift start/end + relaxation timings for that academic year.
  - UI must show the correct shift/relaxation values when staff is added to policy bundle.
  - For efficiency, Senior suggested building/reusing a **view** that takes `EmployeeID` + `AcademicYear` and returns the shift/relaxation outputs, so multiple tables/joins are not needed in UI logic.
