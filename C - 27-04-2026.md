## Topics discussed
### Leave summary: where the data comes from
- Senior explained that **leave summary** (balance view) should be derived by combining:
  - allotted/assigned leave (from policy bundle / leave allocation)
  - used/availed leave (from leave applications)
- Senior gave an example expectation:
  - for casual leave, the UI should show `used` vs `allotted`, and the system should compute/return the **total summary** accordingly.
- The “leave summary” output is expected to support MyHRIS-style display, and it should be based on the staff’s `StaffID` + `AcademicYearID`.

### Leave application: which API to use
- For leave application details, Senior pointed to the need to call the **Get Leave Application** endpoint (staff-wise + academic year-wise).
- Discussion highlighted that when something is confusing, the resolution path is:
  1. confirm the correct endpoint usage
  2. ensure correct parameters (`StaffID`, `AcademicYearID`)
  3. verify response vs database records.

### Policy bundle linkage (HRIS policy bundle)
- Senior clarified that the correct data flow for leave summary/application depends on the staff being properly mapped in **HRIS policy bundle**.
- Specifically, Senior referenced the flow of:
  - **Get applicable bundle by staff ID**
  - then using that policy bundle context for leave-related calculations.
